# Benchmark Method & Compile Flags

Methodology and provenance for this task. The **verified per-run numbers** (geomean, per-shape
speedups, GPU-idle state, full provenance) live in `docs/results.md` and the committed compact
artifact `bench/results_summary.json`, regenerated by each benchmark run — cite those, not any
number hard-coded here.

## Compile flags (symmetric numerics; recorded for provenance)

The candidate is built by `solution/build.py` via `tvm_ffi.cpp.load` (`cuda_files=[kernel.cu]`):

- `extra_cuda_cflags`: `-std=c++17 -O3 -gencode=arch=compute_100,code=sm_100 -lineinfo -D_GLIBCXX_USE_CXX11_ABI=<torch ABI>`
- `extra_cflags`: `-std=c++17 -O3 -D_GLIBCXX_USE_CXX11_ABI=<torch ABI>`
- `extra_ldflags`: torch lib dirs + `-lc10 -lc10_cuda -ltorch_cpu -ltorch_cuda` (for `at::cuda::getCurrentCUDAStream()`).
- **No `--use_fast_math`** and no extra math-mode/arch toggles (the eager reference uses none). The
  split-RoPE kernel uses explicit IEEE-rounded intrinsics — `__fmul_rn`, `__fadd_rn`, `__fsub_rn`,
  `__float2bfloat16_rn`, `__bfloat162float` — so `-O3` cannot contract `a*b±c` into a fused
  multiply-add and reorder the visible bf16 rounding.
- `-lineinfo` is only for Nsight Compute SASS↔source mapping; it does not change codegen.

## Allocation / destination-passing

Output buffers (`outputs[0]`/`outputs[1]`) are **preallocated outside the timed path** and written
in place by `run_candidate` (and by the eager baseline through `call_baseline`). The timed region
excludes input generation, Python setup, JIT build, imports, and output allocation. This is a
**no-output-allocation / destination-passing** contract, not allocation-free execution: the staged
candidate allocates the ATen RMSNorm intermediate (normed q/k) and the eager baseline allocates its
returned tensors; those intermediate allocations exist symmetrically on both sides and are part of
what the eager-replacement comparison measures.

## Timing methodology

`bench/benchmark.py` starts from `standalone_diffusion_benchmark_template.py`. The timing/scoring/
workload policy is unchanged from the template; the only task-local additions are (a) an enriched
`_provenance()` (extra metadata fields, below) and (b) a compact committed evidence artifact
(`bench/results_summary.json`) — neither changes the measurement. Policy:

- Correctness runs before timing; output buffers are poisoned first so unwritten cells / skipped
  kernels are caught.
- GPU time via CUDA events; wall-clock kept only as a diagnostic.
- Per-side inner-loop amplification until a sample is ≥ ~1000 µs (`target_sample_us`).
- A/B interleaving per trial (deterministic order) to cancel clock/thermal drift.
- Each workload runs in an isolated subprocess.
- Reported per workload: median/mean/std/min/p10/p90; headline = equal-weight geometric mean of
  `baseline_median/candidate_median` over production rows.

Settings used (template defaults): `warmup_runs=10`, `num_trials=7`, `inner_iterations_min=1`,
`inner_iterations_max=4096`, `target_sample_us=1000`, `timeout_seconds=600`, isolated runner on.
(`config.toml [benchmark]` lists `inner_iterations_max=2048` / `timeout_seconds=900`; amplification
reaches the ~1000 µs target well below either cap and no workload approaches the timeout, so the
choice does not affect the measurement, and baseline/candidate use identical settings.)

## Provenance recorded per run

`_provenance()` writes, into both `bench/results.jsonl` (raw, local) and `bench/results_summary.json`
(compact, committed): exact command; python/platform; torch + `torch.version.cuda`; GPU name + UUID;
`CUDA_VISIBLE_DEVICES` + `REMOTE_GPU_ID` + `torch.cuda.current_device()`; `nvcc` release line; host
compiler; `tvm_ffi` version; candidate source sha256 (over `solution/{kernel.cu,build.py,candidate.py,__init__.py}`);
baseline commit (from `docs/baseline_source.md`); git commit + dirty flag; `nvidia-smi` before/after;
workload count and trial/iteration settings.

## Candidate entry validation (every call)

`run_candidate` validates inputs on EVERY call and raises `ValueError` before any RMSNorm/kernel
launch when anything is unsupported (TP≠1, non-`torch.nn.RMSNorm`/subclass, eps mismatch, fp32 or
wrong-shape/CPU norm weight, wrong `normalized_shape`, non-bf16 / non-contiguous q/k, odd or
mismatched `head_dim`, bad cos/sin rank/dtype/stride/shape/device, wrong output shape/dtype/
contiguity/device). Validation is not cached by object identity, so in-place mutation of a reused
inputs/outputs object is still re-checked. This is a small per-call metadata cost included in the
timed loop on the candidate side, so the reported eager-replacement speedup is conservative.

## Measurement variability

The eager baseline's small/cross rows (S≈126) are launch/host-overhead bound (~115–220 µs dominated
by per-launch CPU overhead, not tensor size) and thus sensitive to host-CPU contention on the shared
box; the candidate (one GPU-bound kernel per side) is stable (tight p10–p90). Both sides are measured
under identical settings in the same interleaved run, so each A/B ratio is fair; cross-run spread is a
property of the contention-sensitive eager baseline, reported honestly.

## Standalone contract

No sglang import/patch/monkey-patch/install at correctness or benchmark runtime. The baseline
(`baseline/`) and candidate (`solution/`) are task-local; the upstream `apply_split_rotary_emb` eager
fallback was copied (see `docs/baseline_source.md`).
